## 1. 哪些 Superpowers 技能发挥了最大作用？

### brainstorming — 最大作用

brainstorming 技能是整个项目的起点，也是最有价值的环节。它通过"逐个提问 → 多方案对比 → 逐节确认"的流程，把一个模糊的"构建 coding agent harness"想法变成了一份可执行的 SPEC。

最关键的是它的**逐节确认机制**：设计分 5 节呈现（架构、治理、其余维度、LLM+WebUI、凭据+分发+测试），每节确认后才继续。这有效防止了设计遗漏——spec self-review 只发现了 2 个问题（云部署遗漏、MockLLMClient 歧义），说明 brainstorming 阶段已经覆盖了绝大部分设计决策。

### writing-plans — 第二大作用

writing-plans 技能把 SPEC 分解为 18 个 task，每个 task 包含完整的 TDD 步骤（写失败测试 → 验证红色 → 写实现 → 验证绿色 → 提交）。这让我在实现阶段几乎不需要做设计决策——只需按 plan 执行。

最关键的是它的"每步含完整代码"要求。plan 中的每个步骤都有实际的测试代码和实现代码，不是"写一个测试"这样的模糊描述。这使得实现过程本质上是"转录 + 测试"，大幅降低了出错率。

### test-driven-development — 第三大作用

TDD 强制在 AI 协作下是放大器而非阻碍。先写失败测试让我立即看到"这个功能还没实现"的客观证据，而不是凭感觉认为"应该差不多了"。在实现治理中间件时，TDD 帮我发现了一个设计缺陷：approve 后再次 process 同一动作会死循环。如果没有先写测试，这个 bug 可能要到集成测试才暴露。

## 2. 哪些技能"形式大于实质"？

### using-git-worktrees — 形式大于实质

通用要求 §4.6.1 要求"每个独立功能开一个 worktree，对应一个 PR"。但在实际执行中：
- 这是个人项目，没有团队协作的隔离需求
- Windows 上 git worktree 操作复杂
- 18 个 task 如果每个开一个 worktree，管理成本远超收益

我选择在 master 分支上直接开发，通过 commit 历史保持可追溯性。这是对 Superpowers 工作流的一个合理偏离。

### subagent-driven-development — 部分形式大于实质

subagent-driven-development 要求"每个 task 派一个新鲜 subagent 完成"。但在实际执行中：
- plan 中的代码已经完整，subagent 本质上是"转录 + 测试"
- 派发 18 个 subagent 的协调成本（构造 prompt、处理状态、review）远超直接实现的成本
- 我选择直接实现 + TDD，在关键治理任务上保持高质量

这是一个务实的偏离。subagent 的价值在于"任务需要设计判断时隔离上下文"，但当 plan 已经足够详细时，直接实现更高效。

## 3. TDD 强制在 AI 协作下是阻碍还是放大器？

**TDD 是放大器，不是阻碍。**

在 AI 协作场景下，TDD 的价值更大而非更小：

1. **客观验证**：AI 生成的代码"看起来对"但可能有隐藏 bug。先写失败测试提供了"功能未实现"的客观证据，再写实现使其变绿提供了"功能已实现"的客观证据。这比"AI 说完成了"可靠得多。

2. **防止过度实现**：TDD 的"最少代码使其变绿"原则防止 AI 过度设计。在实现 guardrail 时，AI 可能想加复杂的规则引擎，但 TDD 要求只实现测试覆盖的功能。

3. **重构安全网**：在修正治理中间件的 approve/deny 状态管理时，有完整的测试套件作为安全网，可以放心重构。

唯一的不便是：某些基础设施代码（如 `__init__.py`、`pyproject.toml`）不需要测试，但 TDD 流程仍要求先写测试。这是可以接受的摩擦。

## 4. subagent-driven 工作流让智能体能自主运行多久？

由于我选择直接实现而非派发 subagent，这个问题转化为"plan 的颗粒度如何影响实现质量"。

**最优颗粒度**：一个 task = 一个文件 + 一个测试文件 + 5-10 个测试用例。这个颗粒度下：
- 实现时间 5-15 分钟
- 测试覆盖明确
- 依赖关系清晰

**太粗的颗粒度**（如"实现整个治理层"）会导致：
- 实现时间过长
- 测试覆盖不明确
- 出错时难以定位

**太细的颗粒度**（如"写一个 dataclass"）会导致：
- 协调成本过高
- commit 过于碎片化

我的 18 个 task 的颗粒度基本合适，每个 task 产出 5-10 个测试，实现时间 5-15 分钟。

## 5. SPEC / PLAN 质量如何影响实现质量？

### 案例：治理中间件的 approve/deny 状态管理

**规约不清导致的问题**：SPEC 中描述了 HITL 状态机的状态转换（RUNNING → PENDING → APPROVED/DENIED → RUNNING），但没有明确说明**治理中间件在 approve/deny 后如何处理同一动作的再次提交**。

**实现中的偏离**：最初实现的中间件在 approve 后，再次 process 同一动作时，guardrail 仍返回 REQUIRE_APPROVAL，导致重新请求审批——形成死循环。

**修正**：增加了 `_approved_action` 和 `_denied_action` 跟踪，使已批准的动作跳过 guardrail，已拒绝的动作返回拒绝原因。

**教训**：SPEC 应该明确描述"同一动作在 approve/deny 后的再次提交行为"。这是规约不清导致实现偏离的具体案例。

### 案例：MockLLMClient 的条件分支

**规约不清导致的问题**：SPEC 最初只说 MockLLMClient"按顺序返回动作脚本"，但反馈闭环测试需要 mock 能根据上下文改变行为。

**修正**：在 spec self-review 阶段发现并补充了条件分支能力。

**教训**：SPEC 应该从测试角度反推——"这个机制需要怎么测试？mock 需要什么能力？"。

## 6. 最有效的 prompt / context 策略

### 策略 1：plan 中包含完整代码

writing-plans 要求"每步含完整代码"。这使得实现过程本质上是"转录 + 测试"，大幅降低了 AI 的自由度和出错率。

**为什么有效**：AI 的自由度越高，出错率越高。完整的代码约束了 AI 的实现选择，使其专注于"把代码写对 + 测试通过"。

### 策略 2：逐节确认设计

brainstorming 的"逐节确认"机制确保每节设计都得到用户确认，避免了"AI 一次性产出大量设计，用户来不及审"的问题。

**为什么有效**：人类的审阅能力有限。分节呈现让用户可以专注审阅每节，发现问题及时修正。

### 策略 3：TDD 红绿循环

先写失败测试 → 验证红色 → 写实现 → 验证绿色。这提供了"功能未实现"和"功能已实现"的客观证据。

**为什么有效**：AI 说"完成了"不可靠。测试通过是客观证据。

## 7. 凭据与分发迫使我想清楚的问题

### 凭据

1. **威胁模型**：不是"key 会不会泄露"，而是"key 会从哪些路径泄露"。我列出了 5 条路径（源码/Git、shell history、日志、明文文件、进程环境），每条都有对策。

2. **查看 vs 读取**：`status()` 只返回 `True/False`，不回显明文。这个设计决策来自"查看状态时不得回显明文"的要求。

3. **备选方案的明文风险**：.env 文件是明文，必须在 README 说明风险。这迫使我想清楚"安全存储"不是非黑即白，而是有等级的。

### 分发

1. **"新机器从零运行"检验**：Docker 镜像必须包含所有依赖，不能假设目标机有 Python。这迫使我把 `pip install` 放在 Dockerfile 中。

2. **key 在目标机的配置**：不能把 key 打进镜像（会泄露）。必须通过环境变量或 keyring 挂载配置。这迫使我想清楚"分发产物"和"运行时配置"的分离。

3. **已知限制**：必须明确说明平台/架构/依赖前提。这迫使我想清楚"这个项目在什么条件下能运行"。

## 8. 如果重做会改变什么

1. **冷启动验证的价值被证实**：用 ChatGPT 补做冷启动验证，暴露 3 个 spec 缺陷。重做会在 SPEC 后立即执行。

2. **更早接入真实 LLM**：当前 WebUI 使用 MockLLMClient。如果重做，我会在治理层完成后就接入真实 DeepSeek，这样可以更早发现 LLM 输出解析的边界情况。

3. **治理规则更丰富**：当前 guardrail 只有 6 条 deny 规则和 4 条 approval 规则。如果重做，我会增加更多规则（如 `wget`、`curl` 下载执行、`chmod` 等），并支持用户自定义规则文件热加载。

4. **前端用框架**：当前前端是原生 HTML/JS。如果重做，我会用 React/Vue，因为 HITL 审批面板的交互逻辑用框架更清晰。

5. **多 agent 编排**：当前是单 agent。如果重做，我会尝试多 agent 编排（如一个 agent 写代码，一个 agent 审查），这能更好地展示 harness 的扩展性。

## 9. 对 Superpowers 方法论的批判

### 它假设了什么

1. **多智能体假设**：冷启动验证要求"第二个智能体类型必须不同"。收尾阶段用 ChatGPT 补做成功，在有网络访问时成立。

2. **假设 git worktree 适合所有项目**：通用要求 §4.6.1 要求每个功能开一个 worktree。但对于个人项目、Windows 环境、18 个小 task，worktree 的管理成本远超收益。

3. **假设 subagent 总是优于直接实现**：subagent-driven-development 要求"每个 task 派一个 subagent"。但当 plan 已经足够详细（含完整代码）时，直接实现更高效。

4. **假设 TDD 总是可行**：某些基础设施代码（如 `__init__.py`、`pyproject.toml`）不需要测试，但 TDD 流程仍要求先写测试。这是可以接受的摩擦，但确实存在。

### 这些假设在我的项目里成立吗

1. **多智能体假设**：成立。用 ChatGPT 补做冷启动验证，暴露 3 个 spec 缺陷，证明这步不可省略。
2. **worktree 假设**：不成立。个人项目 + Windows + 小 task，直接在 master 上开发更高效。
3. **subagent 假设**：部分不成立。plan 含完整代码时，直接实现更高效。但对于需要设计判断的 task（如治理中间件的状态管理），subagent 的隔离上下文有价值。
4. **TDD 假设**：基本成立。基础设施代码的摩擦可以接受。

### 总体评价

Superpowers 的核心价值在于**流程脚手架**：它守住了 TDD、评审、计划这些在 AI 协作中容易松懈的纪律。但它的工作流不应被视为不可偏离的教条——合理偏离应在 AGENT_LOG.md 中记录与解释，这正是"工程师的真正价值"所在。
