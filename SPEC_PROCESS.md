# SPEC_PROCESS.md

> 记录与 Superpowers 协作生成 spec 与 plan 的过程。

## 1. Brainstorming 关键节点

### 1.1 智能体追问的好问题

在 brainstorming 阶段，智能体通过逐个提问引导设计决策：

1. **"你希望用哪种编程语言？"** — 这个问题迫使我思考生态、测试便利性、WebUI 支持。最终选择 Python，因为 FastAPI + pytest + keyring 的组合最适合快速实现 harness 内核并配套 WebUI。

2. **"你想深入哪个维度？"** — 这个问题让我意识到六个维度不能都浅尝辄止。智能体推荐治理作为重点，理由是"治理天然由代码构成，最契合'机制必须是代码'要求"。我采纳了这个建议，因为 guardrail/sandbox/HITL 状态机确实是最容易用确定性测试验证的。

3. **"你想接入哪个 LLM 供应商？"** — 我选择了 DeepSeek（OpenAI 兼容协议），因为它便宜且 coding 能力足够。智能体确认了可以用 `openai` SDK + 自定义 `base_url` 适配。

4. **"WebUI 承担什么职责？"** — 这个问题让我明确了 WebUI 不只是监控面板，而是任务对话 + HITL 审批的完整交互界面。

### 1.2 修正原设想的决策

- **架构方案**：我原本倾向于插件架构（方案 C），认为扩展性更重要。智能体推荐单体中间件（方案 A），理由是"课程项目深度优先，插件架构会稀释深度"。我接受了这个建议，事后证明正确——治理中间件模式让所有机制都是独立可测的纯函数。

- **分发形态**：我原本想同时做 Docker + PyPI。智能体建议只做 Docker，因为"YAGNI，Docker 已足够覆盖'新机器从零运行'检验"。我采纳了。

## 2. 至少 3 轮关键迭代

### 迭代 1：架构方案选择

**智能体提出**：3 种架构方案（单体中间件 / 微服务 / 插件），推荐单体中间件。

**我的处理**：我原本倾向插件架构，但智能体的理由"深度优先，插件架构会稀释深度"说服了我。我选择了单体中间件，并要求治理作为中间件链中的核心环节。

**决策**：采纳智能体推荐，方案 A。

### 迭代 2：MockLLMClient 的条件分支能力

**问题**：在设计 LLM 抽象层时，最初只设计了"按顺序返回动作脚本"的 MockLLMClient。但在设计反馈闭环测试时发现：如果 mock LLM 不能根据上下文改变行为，就无法测试"反馈驱动自我修正"。

**我的处理**：要求在 MockLLMClient 中增加条件分支——支持 `{"if_context_contains": "失败关键词", "action": Action}` 格式，使 mock 能检查上下文中是否包含特定反馈字符串。

**决策**：修正了设计，在 SPEC §3.1.2 和 PLAN Task 3 中补充了条件分支能力。

### 迭代 3：治理中间件的状态管理

**问题**：最初设计的 GovernanceMiddleware 在 approve/deny 后，再次 process 同一动作会重新触发 guardrail 的 REQUIRE_APPROVAL，导致死循环。

**我的处理**：在实现阶段发现这个问题，修正了中间件逻辑——增加 `_approved_action` 和 `_denied_action` 跟踪，使已批准的动作跳过 guardrail 直接进入 sandbox，已拒绝的动作返回拒绝原因。

**决策**：这是实现中发现并修正的设计缺陷，记录在 AGENT_LOG.md 中。

## 3. AI 建议 vs 我的决策

### 采纳的 AI 建议

| 建议 | 理由 |
|------|------|
| Python 作为主语言 | 生态丰富，FastAPI + pytest + keyring 组合最优 |
| 治理作为重点维度 | 天然由代码构成，最契合"机制必须是代码"要求 |
| 单体中间件架构 | 深度优先，避免插件架构稀释深度 |
| 只做 Docker 分发 | YAGNI，Docker 已足够 |
| 原生 HTML/JS 前端 | 无需构建工具，WebUI 非重点维度 |

### 我推翻或修正的

| AI 建议 | 我的修正 | 理由 |
|---------|----------|------|
| MockLLMClient 只按顺序返回 | 增加条件分支能力 | 否则无法测试反馈闭环 |
| 治理中间件简单串联 | 增加 approved/denied 状态跟踪 | 避免approve后死循环 |
| spec 未包含云部署 | 补充 §7.3 Render 部署 | 通用要求 §5 要求线上 URL |

## 4. 反思

### Brainstorming 做得好的地方

- **逐节确认机制**：设计分 5 节呈现，每节确认后才继续，有效防止了设计遗漏
- **多方案对比**：提出 3 种架构方案并给出推荐理由，帮助我做决策而非被动接受
- **spec self-review**：发现了 2 个实质性问题（云部署遗漏、MockLLMClient 歧义）

### Brainstorming 让我不满的地方

- **未追问具体危险动作列表**：智能体没有追问我想拦截哪些具体命令，导致规则列表是通用的。实际实现时我自行补充了 `rm -rf /`、`drop database`、`curl | sh` 等。
- **未讨论测试覆盖度**：brainstorming 阶段没有讨论每个机制需要多少测试用例，导致 PLAN 阶段才确定测试数量。
- **冷启动验证未执行**：通用要求 §4.5 要求用"陌生"智能体冷启动试运行，但由于环境限制（只有一个编码智能体可用），这一步未执行。这是规约质量验证的一个缺口。

## 5. 冷启动验证（§4.5）

通用要求 §4.5 要求用与主开发智能体不同的 agent，在不提供对话历史的前提下，仅凭 SPEC + PLAN 尝试实现 1-2 个 task。

**执行情况**：由于当前环境只有一个编码智能体（opencode + glm-5.2），无法满足"第二个智能体类型必须不同"的要求。这一步未能执行。

**替代措施**：通过 spec self-review 和 PLAN self-review 部分弥补——self-review 发现了 2 个实质性问题（云部署遗漏、MockLLMClient 歧义），说明 spec 质量验证仍有价值。

**教训**：冷启动验证是规约工作中最关键的"客观证据"，未来应确保有第二个智能体可用。
