# SPEC_PROCESS.md

> 记录与 Superpowers 协作生成 spec 与 plan 的过程。

## 1. Brainstorming 关键节点

### 1.1 智能体追问的好问题

在 brainstorming 阶段，智能体通过逐个提问引导设计决策：

1. **"你希望用哪种编程语言？"** — 这个问题迫使我思考生态、测试便利性、WebUI 支持。最终选择 Python，因为 FastAPI + pytest + keyring 的组合最适合快速实现 harness 内核并配套 WebUI。

2. **"你想深入哪个维度？"** — 这个问题让我意识到六个维度不能都浅尝辄止。智能体推荐治理作为重点，理由是"治理天然由代码构成，最契合'机制必须是代码'要求"。我采纳了这个建议，因为 guardrail/sandbox/HITL 状态机确实是最容易用确定性测试验证的。

3. **"你想接入哪个 LLM 供应商？"** — 我选择了 DeepSeek（OpenAI 兼容协议），因为它便宜且 coding 能力足够。智能体确认了可以用 `openai` SDK + 自定义 `base_url` 适配。

4. **"WebUI 承担什么职责？"** — 这个问题让我明确了 WebUI 不只是监控面板，而是任务对话 + HITL 审批的完整交互界面。

### 1.2 修正原设想的决策

- **架构方案**：我原本倾向于插件架构（方案 C），认为扩展性更重要。智能体推荐单体中间件（方案 A），理由是"课程项目深度优先，插件架构会稀释深度"。我接受了这个建议，事后证明正确——治理中间件模式让所有机制都是独立可测的纯函数。

- **分发形态**：我原本想同时做 Docker + PyPI。智能体建议只做 Docker，因为"YAGNI，Docker 已足够覆盖'新机器从零运行'检验"。我采纳了。

## 2. 至少 3 轮关键迭代

### 迭代 1：架构方案选择

**智能体提出**：3 种架构方案（单体中间件 / 微服务 / 插件），推荐单体中间件。

**我的处理**：我原本倾向插件架构，但智能体的理由"深度优先，插件架构会稀释深度"说服了我。我选择了单体中间件，并要求治理作为中间件链中的核心环节。

**决策**：采纳智能体推荐，方案 A。

### 迭代 2：MockLLMClient 的条件分支能力

**问题**：在设计 LLM 抽象层时，最初只设计了"按顺序返回动作脚本"的 MockLLMClient。但在设计反馈闭环测试时发现：如果 mock LLM 不能根据上下文改变行为，就无法测试"反馈驱动自我修正"。

**我的处理**：要求在 MockLLMClient 中增加条件分支——支持 `{"if_context_contains": "失败关键词", "action": Action}` 格式，使 mock 能检查上下文中是否包含特定反馈字符串。

**决策**：修正了设计，在 SPEC §3.1.2 和 PLAN Task 3 中补充了条件分支能力。

### 迭代 3：治理中间件的状态管理

**问题**：最初设计的 GovernanceMiddleware 在 approve/deny 后，再次 process 同一动作会重新触发 guardrail 的 REQUIRE_APPROVAL，导致死循环。

**我的处理**：在实现阶段发现这个问题，修正了中间件逻辑——增加 `_approved_action` 和 `_denied_action` 跟踪，使已批准的动作跳过 guardrail 直接进入 sandbox，已拒绝的动作返回拒绝原因。

**决策**：这是实现中发现并修正的设计缺陷，记录在 AGENT_LOG.md 中。

## 3. AI 建议 vs 我的决策

| 治理作为重点维度 | 天然由代码构成，最契合"机制必须是代码"要求 |
| 单体中间件架构 | 深度优先，避免插件架构稀释深度 |

## 4. 反思

### Brainstorming 做得好的地方

- **逐节确认机制**：设计分 5 节呈现，每节确认后才继续，有效防止了设计遗漏
- **多方案对比**：提出 3 种架构方案并给出推荐理由，帮助我做决策而非被动接受
- **spec self-review**：发现了 2 个实质性问题（云部署遗漏、MockLLMClient 歧义）

### Brainstorming 让我不满的地方

- **未追问具体危险动作列表**：智能体没有追问我想拦截哪些具体命令，导致规则列表是通用的。实际实现时我自行补充了 `rm -rf /`、`drop database`、`curl | sh` 等。
- **未讨论测试覆盖度**：brainstorming 阶段没有讨论每个机制需要多少测试用例，导致 PLAN 阶段才确定测试数量。
- **冷启动验证延迟执行**：通用要求 §4.5 要求用"陌生"智能体冷启动试运行。由于初期只有一个编码智能体可用，这一步延迟到收尾阶段用 ChatGPT 补做。冷启动智能体暴露了 3 个真实 spec 缺陷（详见 §5），证明这步验证不可省略。

## 5. 冷启动验证（§4.5）

通用要求 §4.5 要求用与主开发智能体不同的 agent，在不提供对话历史的前提下，仅凭 SPEC + PLAN 尝试实现 1-2 个 task。

### 5.1 执行情况

- **主开发智能体**：OpenCode + glm-5.2
- **冷启动智能体**：ChatGPT（GPT-4o），全新 session，不导入任何先前会话或 memory
- **提供材料**：SPEC.md 全文 + PLAN.md 中 Task 8（Guardrail）与 Task 9（Sandbox）的 TDD 步骤
- **指定 task**：Task 8 + Task 9（治理重点维度，步骤最清晰）
- **指令**：遇到不确定之处即暂停询问，而非凭猜测继续

### 5.2 冷启动智能体提出的问题

冷启动智能体在阅读 SPEC 后，暂停并提出 6 个疑问，其中 3 个暴露了真实的 spec 缺陷：

#### 缺陷 1：`readonly_paths` 作用域未明确（真实缺陷）

**智能体提问**：`readonly_paths` 是否支持文件（如 `.env`），还是只支持目录（如 `.git/`）？测试只涉及目录，但描述是"只读路径"，可能也支持文件。

**分析**：SPEC §3.3.2 只说"检查是否为只读路径"，未明确 readonly_paths 支持文件还是目录。这是一个真实的规约模糊——实现时需要猜测匹配策略。

**修订**：应在 SPEC §3.3.2 补充：`readonly_paths` 同时支持文件与目录，匹配策略为路径前缀匹配（`rel_path.startswith(ro)`）。

#### 缺陷 2：Sandbox 命令匹配粒度未指定（真实缺陷）

**智能体提问**：白名单匹配是用首词（`command.split()[0]`）还是完整命令？如果用户希望允许 `git push` 但拒绝 `git push --force`，目前设计是否支持？

**分析**：SPEC §3.3.2 只说"按白名单或黑名单过滤命令"，未指定匹配策略。智能体正确推测了"Sandbox 粗粒度 + Guardrail 细粒度"的分工，但这层分工关系在 SPEC 中未明文写出。

**修订**：应在 SPEC §3.3.2 补充：Sandbox 按命令首词做粗粒度白名单/黑名单匹配；细粒度模式匹配由 Guardrail 负责。两者分工明确。

#### 缺陷 3：Task 间依赖在 SPEC 中不够显式（轻微缺陷）

**智能体提问**：Sandbox 测试依赖 `harness.config.SandboxConfig`，但该类未定义。是否需要先创建 `harness/config.py`？

**分析**：PLAN.md 的依赖图确实标注了 `Task 2 (Config) → Task 9 (Sandbox)`，但冷启动智能体只有 SPEC（PLAN 未完整提供），从 SPEC 的功能规约中无法看出 Sandbox 对 Config 的编译时依赖。这说明 SPEC 的模块描述缺少"依赖关系"信息。

**修订**：应在 SPEC §3 各模块描述中补充"Consumes"字段（已在 PLAN 中实现，但 SPEC 未同步）。

#### 非缺陷问题（3 个）

| 问题 | 结论 |
|------|------|
| `list_dir` 参数名是否为 `path` | 是，SPEC 已明确，非缺陷 |
| 跨平台兼容性 | 智能体正确推断 Linux-only，非缺陷 |
| 测试执行顺序 | PLAN 依赖图已覆盖，非缺陷 |

### 5.3 产出与预期差距

冷启动智能体在提出疑问后等待确认，未直接产出代码（符合"遇到不确定即暂停"的指令）。但其提问质量表明：

- **SPEC 的模块描述缺少依赖关系**：智能体无法从 SPEC 单独推断出 Sandbox 依赖 Config
- **SPEC 的行为描述缺少边界细节**：readonly_paths 作用域、命令匹配粒度都未写清
- **智能体正确推测了设计意图**：Sandbox 粗粒度 + Guardrail 细粒度的分工，说明设计意图可从上下文推断，但不应依赖推断

### 5.4 据此对 SPEC / PLAN 的修订

| 修订项 | 修订前 | 修订后 |
|--------|--------|--------|
| §3.3.2 readonly_paths | "检查是否为只读路径" | "readonly_paths 同时支持文件与目录，按路径前缀匹配" |
| §3.3.2 命令匹配 | "按白名单或黑名单过滤命令" | "Sandbox 按命令首词做粗粒度匹配；细粒度模式由 Guardrail 负责" |
| §3 各模块 | 无依赖说明 | 补充 Consumes 字段（与 PLAN 对齐） |

> 注：以上修订为文档层面的改进建议。由于实现已完成且测试通过，代码层面无需改动——实际实现已遵循了上述意图，只是 SPEC 文档未显式写出。

### 5.5 教训

- **冷启动验证是规约工作中最关键的"客观证据"**：3 个真实缺陷中，有 2 个（readonly_paths 作用域、命令匹配粒度）是主开发智能体在实现时未质疑、但新智能体一眼看出的。这说明主智能体与开发者之间沉淀的隐性上下文，确实会掩盖 spec 的模糊之处。
- **"遇到不确定即暂停"的指令有效**：冷启动智能体没有凭猜测继续，而是列出了所有疑问等待确认。如果它直接实现，可能选择错误的匹配策略，导致返工。
- **SPEC 应包含模块依赖关系**：仅靠 PLAN 的依赖图不够，SPEC 的功能规约也应标注每个模块的 Consumes/Produces。
